## 1. THE ALPHAPROOF NEXUS ARCHITECTURE

The AlphaProof Nexus framework is a decentralized, reinforcement-learning-guided agentic system designed for formal reasoning, code generation, and proof search. It utilizes evolutionary game-play loops, a shared population database, Elo rating feedback, and strict compiler-in-the-loop verification (the Ralph Loop).

### 1.1 THE RALPH LOOP (PROVER AGENT)
The Basic Prover Agent (A) operates via the "Ralph Loop," an iterative, multi-turn reasoning and editing protocol:
- **Search-Replace Editing**: The LLM is restricted from rewriting entire files. It must output surgical search-replace blocks (similar to diffs) to incrementally mutate code or proof states.
- **Compiler Feedback**: After each edit, the code/proof is run directly through a compiler or tactic prover (e.g., Lean 4). Any warning or error traceback is fed directly back into the LLM context in the subsequent turn.
- **Sorry-Free Constraint**: In Lean, a proof must be complete without the "sorry" tactic (which denotes an unproven stub). In the Ralph Loop, the agent's absolute target is the recursive elimination of all "sorry" placeholders.

### 1.2 THE POPULATION DATABASE (EVOLUTIONARY AGENT)
The Full-Featured Evolutionary Agent (D) coordinates a population of sub-agents via a shared Population Database:
- **Global Goal Caching**: Eliminates duplicate paths. A deep cryptographic hash of the formal context (e.g., imports, open namespaces, local variables) and the current goal is calculated. If the goal is cached as proven, the solution is retrieved immediately; if cached as unprovable, the branch is aborted.
- **Population Registry**: Stores candidate proof sketches, tactics, and intermediate strategies generated by the swarm.

### 1.3 ELO-BASED RATING (RATER AGENTS)
Rather than absolute numeric grading, sketches are compared pairwise by Rater Agents (using high-speed models like Gemini Flash):
- **Plackett-Luce Model**: Pairs of sketches are ranked against each other based on logical depth and brevity.
- **Gibbs Sampling**: The pairwise rankings are processed using Gibbs sampling to compute the posterior strength distribution of each candidate.
- **Elo Ratings**: These strengths are converted to standardized Elo ratings, enabling the MCTS engine to identify which branches are leading, stagnant, or declining.

### 1.4 EVOLUTIONARY SELECTION (P-UCB SEARCH)
The MCTS engine explores the proof tree using the Predictor Upper Confidence Bound (P-UCB) formula to coordinate expansions:
- **P-UCB Selection Rule**:
  $$P-UCB_i = Q_i + c \cdot P_i \cdot \frac{\sqrt{N}}{1 + N_i}$$
  Where:
  - $Q_i$ is the average reward score of the node (based on compiler and rater feedback).
  - $P_i$ is the prior probability of selecting the branch, derived from the node's Elo rating.
  - $N_i$ is the number of visits to the branch.
  - $N$ is the total visits of the parent.
  - $c = 0.2$ is the exploration constant.
- **Guillotine Protocol (Pruning)**: The top 64 sketches are retained. Low-performing branches are aggressively pruned and evicted from the Population Database to conserve compute resources.

### 1.5 SAFEVERIFY GATES
A pre-flight validation gate that intercepts LLM outputs to prevent:
- **Axiom Injection**: The agent attempting to declare a theorem as true by injecting custom, unverified axioms (equivalent to mock stubs).
- **Environment Exploits**: Code injection in compiler command loops.
- **EVOLVE-BLOCK / EVOLVE-VALUE Markers**: Defining segments of code/state that the model is allowed to mutate while locking the remaining structure as immutable.
