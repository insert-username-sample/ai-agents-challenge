# PROPRIETARY TECHNOLOGY VISION: CLAUSELY V1.5 COMPILER & DYNAMIC WORKSPACES

---

## 1. The v1.5 Vision: A Self-Compiling User Interface

Clausely v1.5 is a vision-stage direction for legal agent interfaces: **transitioning from static dashboards to on-the-go UI compilation**.

In the current alpha, the document editor, timeline viewer, and case playbooks exist as hardcoded static components. In the v1.5 vision, the user interface compiles itself in real time based on what the strategist needs, what verified legal context permits, and what the lawyer commands. Future System 5 Melaquera may audit these compiled workspaces, but it is not part of the current strategist implementation cycle.

```
 [Natural Language Input] ──> [Intent Classifier (Gemma 4)] ──> [UI Spec Generator]
                                                                        │
                                                                        ▼
                                                             [AST Compiler Engine]
                                                                        │
                                                                        ▼
                                                             [Dynamic UI Rendering]
                                                              (JSX / Tailwind CSS)
```

For example, typing:
- *"Calculate stamp duty for a 25 Lakh rent agreement in Nagpur"* dynamically generates a custom numeric stamp duty calculator widget.
- *"Compare Bombay HC Nagpur Bench and Principal Seat formatting rules"* compiles a side-by-side comparative document metric card.
- *"Draw Smt. Vidya Khobrekar's maternal family tree"* compiles an interactive SVG genealogy tree utilizing verified community acceptance nodes.

## 2. On-the-Go UI Generation Pipeline

The runtime compilation is driven by a persistent natural language input field at the bottom of the workspace, running through a three-stage generation loop:

### 1. Intent Classification (Latency Target: < 200ms)
*   **Model**: **Gemma 4 E2B** (deployed locally on-device for zero-latency indexing).
*   **Action**: Evaluates user input against structured intent categories (VisualisationRequest, InteractiveToolRequest, InformationLookup, DocumentAction, AgentTrigger).

### 2. UI Specification Generation (Latency Target: < 3s)
*   **Model**: **Gemma 4 E4B** or **Step-Fun 2.7 Flash**.
*   **Action**: Generates a structured UI specification payload. The spec defines the component type, the raw data schema, binding parameters from the current legal matter context, and strict Clausely design tokens.

### 3. Dynamic Rendering & Sandboxed Compilation (Latency Target: < 10s)
*   **Model**: **React JSX Compiler** + **Tailwind CSS Engine**.
*   **Action**: Compiles the specification directly into sandboxed React JSX. The engine injects the component into the active viewport, binding to live data states while applying strict CSP (Content Security Policy) protections to prevent unauthorized script execution.

## 3. The Legal AST v1.5 Schema Specification

To ensure generated UI components are 100% grounded in legal facts, we define a structured **Legal Abstract Syntax Tree (AST)**. The compiler reads this AST to determine which interactive components to render.

### TypeScript Definition (`packages/legal-ast/ast.ts`)

```typescript
export type LegalNodeType = 
  | 'IntakeBrief' 
  | 'PrecedentCitation' 
  | 'ProceduralMilestone' 
  | 'MaternalLineageNode' 
  | 'VigilanceInquiry';

export interface BaseLegalNode {
  id: string;
  type: LegalNodeType;
  version: string;
  isGrounded: boolean;
  validationSource?: string;
}

export interface PrecedentCitationNode extends BaseLegalNode {
  type: 'PrecedentCitation';
  citation: string;
  rulingCourt: string;
  holdingRatio: string;
  isBinding: boolean;
}

export interface MaternalLineageNode extends BaseLegalNode {
  type: 'MaternalLineageNode';
  generationDepth: number;
  communityAcceptanceVerified: boolean;
  vigilanceCellReportRef?: string;
}

export interface LegalAST {
  root: BaseLegalNode;
  nodes: Record<string, BaseLegalNode>;
}
```

## 4. AST-Driven Component Compilation

The AST compiler maps nodes to dynamic interface templates:

### 1. Maternal Lineage Node
*   **AST Signal**: `MaternalLineageNode` detected with `generationDepth = 3`.
*   **Compiled UI**: Compiles an interactive, 3-generation maternal genealogy chart. This chart allows advocates to map Smt. Vidya Khobrekar's maternal relatives, highlighting those possessing valid Scheduled Caste Mahar certificates to present to the Caste Scrutiny Committee.

### 2. Precedent Citation Node
*   **AST Signal**: `PrecedentCitation` detected with `citation = "Rameshbhai Dabhai Naika"`.
*   **Compiled UI**: Compiles a dual-view split panel. The left side displays the drafted pleading; the right side embeds an interactive, verified frame highlighting the *Ratio Decidendi* of *Rameshbhai Dabhai Naika* (2012) 3 SCC 400.

### 3. Future System 5 Audit Node
*   **AST Signal**: `ProceduralMilestone` detected with a future System 5 audit action.
*   **Compiled UI**: Compiles a warning banner that explains what failed, what could have happened, why the failure occurred, how the failure probability can be reduced, and how the grounded win/pass estimate changes. Until System 5 is explicitly implemented, current strategist UI must rely on per-step grounding outputs from the seven canonical strategist agents.
